forgotten paths revival

$ april 9th, 2026

Starting Over

About 8 months ago, my friend and I submitted Forgotten Paths to the GMTK 2025 Game Jam. This is one of the biggest game jams ever held, with over 9,500 submissions (at the time of writing). We were really happy with how it turned out, and so were the people who played it. We ranked very well, which felt incredible given the scale of the competition.

The feedback we got was really encouraging and gave us a lot of ideas. So we decided to take Forgotten Paths further and turn it into a full Steam release. We decided to start completely from scratch and rebuild it to support future development as opposed to try and fix our buggy and spaghetti code from the jam. The core concept is the same, but everything under the hood is being rebuilt properly to support all the features we want to add.

This is the first devlog for the revamped Forgotten Paths. I want to document what we're building, how we're thinking about it, and what's changed from the original. If you wanted to play the jam version to see where we started, you can find it here.

The New Structure

The biggest lesson from the jam version was that clean, extensible code makes everything easier. The jam gave us 96 hours, so shortcuts were taken. This time we don't have that excuse, and we're building the backend right from the start.

At the core of the new version is the BattleEngine. It's the main backend manager that drives everything. It handles the game loop, processes tiles and entities each tick, tracks the player's command budget, and determines win/lose conditions. The tick order is always tiles → enemies → player, and every tick returns an array of GameEvents instead of directly mutating the scene. This means the UI and logic are fully decoupled. The engine doesn't care how anything gets displayed; it just processes state and spits out events. There is a small code snippet below that illustrates this approach. The renderer can respond to those events however it wants, or ignore them entirely. This also enables us to run the engine in a headless mode for the Simulator, which I'll talk about later.

The grid itself is managed by the GridManager, which uses two dictionaries instead of 2D arrays. One mapping positions to tiles, and one mapping positions to entities. We found that Dictionary-based grids are just more flexible than arrays for this kind of puzzle game. Sparse maps, irregular shapes, fast lookups all work naturally. The tile grid is set once when a map loads and never changes. The entity grid is the live, mutable state.

Maps themselves are defined by MapData resources. These are plain Godot .tres files that describe grid dimensions, tile spawns, entity spawns, the player's starting position, and a couple of constraints for puzzle design like the command budget and max loop length. This gets loaded by the GridManager at the start of each level and used to populate the tile and entity dictionaries. 

# tick() is the core loop. call this once per game step.
# tick order: tiles -> enemies -> player
func tick() -> Array[GameEvent]:
	if is_terminal:
		Clogger.warn("ticking after a terminal event")
		return []
		
	var events: Array[GameEvent] = []

	events.append_array(_process_tick_tiles())
	if _check_terminal(events): return events

	events.append_array(_process_entities())
	if _check_terminal(events): return events

	var cmd: Command = player.current_command()
	events.append_array(_process_player())
	if _check_terminal(events): return events

	# deduct the cost of the command that just ran
	if remaining_budget >= 0 and cmd != null:
		remaining_budget -= cmd.cost
		if remaining_budget <= 0:
			remaining_budget = 0
			events.append(GameOverEvent.new(GameOverEvent.Reason.BUDGET_EXHAUSTED))

	return events

The Command Loop

The core mechanic of Forgotten Paths is the command loop. The player doesn't directly control their character tick by tick. Instead, they set up a sequence of commands that the player character then executes repeatedly in a cycle. Getting that loop right is the puzzle.

In the new codebase this is modeled with a proper Command base class that each command type extends. Right now we have MoveCommand, RotateCommand, and WaitCommand in an attempt to mimic the behavior of the original jam version. Each one has an execute() method that takes an entity and the grid and returns a list of GameEvents. This pattern makes adding new commands really easy. You just create a new class, implement execute(), and it plugs right into the system without touching anything else. This was an intentional design choice to make future expansion easier. We have a long list of commands we want to add eventually, and this structure means we can do that without changing and breaking existing code.

MoveCommand supports all four directions and is aware of the player's current rotation, so a command that says "move up" might actually move the player right if they've been rotated 90 degrees clockwise. RotateCommand is what changes that rotation state. The WaitCommand is a simple no-op which will be useful for timing-based puzzles when we get there. Each command also has a cost field that gets deducted from the map's budget when it runs, which is how we can make some actions more expensive than others.

One other important design choice we made is that Command instances take an Entity as a parameter. This allows us to use the same command system for enemy behavior as well. If we want to add an enemy that moves in a pattern, we can just give it a command loop like the player's and have it execute that each tick. This keeps everything consistent and reusable. If we need special handling per entity type, we can always check the entity's class in the command's execute method and adjust behavior accordingly. The code snippet below shows how this works for MoveCommand. 

# Adding a new command is as simple as extending Command and implementing execute().
# Nothing else in the engine needs to change.

extends Command
class_name MoveCommand

var direction: Consts.Direction

func execute(entity: Entity, grid: GridManager) -> Array[GameEvent]:
	var target: Vector2i
	if entity is Player:
        # rotation is applied here, so the loop's directions are relative to
        # the player's current facing - not the world grid.
		var actual_dir = RotationHelper.rotate_direction(direction, entity.current_rotation)
		target = GridHelper.get_movement_in_direction(entity.position, actual_dir)
	else:
		target = GridHelper.get_movement_in_direction(entity.position, direction)
	return grid.move_entity(entity, target)

# factory methods keep loop construction readable at the call site
static func up()    -> MoveCommand: return MoveCommand.new(Consts.Direction.UP)
static func down()  -> MoveCommand: return MoveCommand.new(Consts.Direction.DOWN)
static func left()  -> MoveCommand: return MoveCommand.new(Consts.Direction.LEFT)
static func right() -> MoveCommand: return MoveCommand.new(Consts.Direction.RIGHT)

Tiles & Entities

Both tiles and entities follow the same design philosophy: a base class that defines the interface, and subclasses that override specific behavior. The BaseTile class has two overridable methods: is_passable() and on_entered(). Most tiles are passable and do nothing when stepped on, but some override those methods to create walls, traps, portals, etc.

Currently we have BlockedTile (the rocks and walls from the jam), PortalTile (teleports the entity to its paired portal on entry), and TickTile (a base class for tiles that need to fire behavior every tick, like traps). The TickTile is a base class for tiles that need to fire behavior every tick, like traps. Most tiles don't need that, but this allows us to support more complex tile behavior without bloating the base class with a bunch of empty methods.

On the entity side, BaseEnemy extends the Entity base class and adds the concept of being alive and an on_entered() callback for when the player walks into them. The first concrete enemy type is StaticEnemy which just stands there and dies when the player bumps into it. That's the only enemy type in the jam game too, but now the system is set up to support enemies that move, have patterns, react to rotations, etc. We'll get there.

The Simulator

This is probably my favorite part of the new codebase. The Simulator is a tool that wraps the BattleEngine and lets us test command loops programmatically. You give it a map and a loop, it runs the engine until there's a win, loss, or timeout, and returns a SimResult describing what happened.

The really powerful function is simulate_all(). It generates every possible command loop up to the map's max_loop_length using base-N enumeration over the command palette, runs each one, and collects every loop that results in a win. This lets us verify that a given map is actually solvable, and find all the solutions. If a map has 50 winning loops when we only intended there to be 2 or 3, that tells us the puzzle is too loose and needs more constraints. If it has zero, we broke something.

It's essentially a built-in puzzle validator, and it means we can design levels with a lot more confidence than we had during the jam, where we were mostly just playtesting by hand. It's going to be indispensable as the puzzle complexity grows. 

# simulate_all() brute-forces every possible loop up to max_loop_length.
# returns only the winning SimResults so we can audit solution counts per map.
func simulate_all(max_ticks: int = 100) -> Array[SimResult]:
    var max_length: int = _engine.grid._loaded_map.max_loop_length
    var palette: Array[Command] = [
        MoveCommand.up(), MoveCommand.down(),
        MoveCommand.left(), MoveCommand.right(),
        RotateCommand.r90cc(), RotateCommand.r270cc(), RotateCommand.r180(),
    ]

    var wins: Array[SimResult] = []
    for length in range(1, max_length + 1):
        var combo_count: int = int(pow(palette.size(), length))
        for i in range(combo_count):
            var loop: Array[Command] = _index_to_loop(i, palette, length) # base-N enumeration to get the loop for this combo index
            var result: SimResult = simulate(loop, max_ticks)
            result.loop = loop
            if result.outcome == SimResult.Outcome.WIN:
                wins.append(result)

    return wins

What's Next

The backend is in a solid place. The BattleEngine ticks correctly, the Simulator validates maps, entities and tiles are extensible, and the event system keeps everything decoupled. What doesn't exist yet is anything visual. There is no renderer, no UI, no actual Godot scene to run it in. That's the next big chunk of work.

Beyond that, we have a long list of things to build out based on the jam feedback and our own ideas: more enemy types, more tile types, a proper level editor, a UI for building the command loop, and eventually a full campaign. There's a lot to do, but for the first time it actually feels like the foundation is solid enough to build on top of.

I'll keep posting these as progress gets made. If you want to follow along or played the jam version and have thoughts, feel free to reach out.
